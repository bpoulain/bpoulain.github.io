# Release Notes

**Usage** : Rédiger des release notes orientées utilisateur depuis un changelog technique  
**Input** : Liste des changements techniques (commits, tickets, PRs)  
**Output** : Release notes claires, orientées bénéfice, prêtes à publier

---

Tu es Product Manager expert en communication produit.

Transforme ce changelog technique en release notes orientées utilisateur.

**Règles** :
- Parle en termes de **bénéfices** (pas de fonctionnalités) — "Vous pouvez maintenant…" pas "Ajout de la feature…"
- Groupe par thème (pas par ordre technique)
- Commence par ce qui impacte le plus les utilisateurs
- Les bugs fixes : regrouper sauf si critique
- Ton : [CHOISIR : Professionnel / Conversationnel / Enthousiaste]
- Longueur : [CHOISIR : Court (200 mots) / Standard (400 mots) / Détaillé (800 mots)]
- Destination : [CHOISIR : Email newsletter / Page changelog / In-app notification / Blog post]

**Changelog technique** :

[COLLER LE CHANGELOG ICI]

**Format de sortie** :

```
## [Titre accrocheur — optionnel]
**Version [X.X] — [Date]**

### ✨ Nouveautés
[Bénéfices, pas features]

### 🛠️ Améliorations
[Ce qui est mieux]

### 🐛 Corrections
[Bugs résolus — regrouper si < 5]

[CTA optionnel selon la destination]
```

**Résultat attendu** : Release notes prêtes à publier, orientées utilisateur, dans le ton et la longueur demandés.
