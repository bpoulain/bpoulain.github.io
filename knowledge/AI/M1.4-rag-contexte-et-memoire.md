# M1.4 — RAG : quand l'IA connaît votre contexte

> Le RAG (Retrieval Augmented Generation) est probablement la technique IA la plus utile pour un product designer ou PM à comprendre en 2024-2025. Elle explique comment des produits comme Notion AI, les chatbots de documentation ou les assistants personnalisés peuvent répondre avec précision à partir de **vos** données.

---

## 1. Théorie

### Définition

Le RAG est une technique qui combine deux opérations distinctes : **Retrieval** (recherche de documents pertinents dans une base de connaissances) et **Generation** (génération d'une réponse par un LLM en utilisant ces documents comme contexte). Au lieu de se fier uniquement à ce qu'il a mémorisé lors de son entraînement, le modèle consulte d'abord des sources externes, puis formule sa réponse à partir de ces sources.

### Modèles mentaux

**Modèle 1 : L'assistant avec accès aux notes**
Imagine un collaborateur expert mais avec une mémoire courte. Sans RAG, il répond de sa mémoire générale — souvent utile, parfois inexact. Avec RAG, avant de répondre, il consulte les notes de la réunion, la documentation produit, ou les emails pertinents. Sa réponse est ancrée dans **ton** contexte, pas dans des généralités.

**Modèle 2 : La recherche sémantique comme clé**
La partie "Retrieval" ne fonctionne pas comme une recherche par mots-clés (Google), mais par similarité sémantique. "Comment réinitialiser mon mot de passe ?" trouvera le document intitulé "Récupération d'accès au compte" même si aucun mot ne correspond exactement — parce que les deux phrases ont la même intention.

### Schéma conceptuel

```
Question utilisateur
        ↓
  Embedding de la question
  (transformation en vecteur numérique)
        ↓
  Recherche vectorielle dans la base de connaissances
  (comparaison de similarité avec tous les documents indexés)
        ↓
  Top K documents les plus pertinents récupérés
        ↓
  Injection dans le prompt
  "Contexte : [documents récupérés]
   Question : [question utilisateur]
   Réponds uniquement à partir du contexte fourni."
        ↓
  LLM génère la réponse
        ↓
  Réponse ancrée dans les sources
```

### Analogie

"RAG, c'est comme un chatbot qui peut consulter ta bibliothèque avant de te répondre. Sans RAG, il parle de mémoire. Avec RAG, il feuillette tes livres, trouve les passages pertinents, et te répond avec des références précises."

## 2. Compréhension

### 5 questions de vérification

1. Quelle est la différence fondamentale entre un LLM "nu" et un LLM avec RAG ?
2. Pourquoi la recherche vectorielle est-elle meilleure que la recherche par mots-clés pour le RAG ?
3. Dans quel cas le RAG n'est-il pas la bonne solution (et qu'utiliser à la place) ?
4. Qu'est-ce qu'un "chunk" et pourquoi la taille des chunks affecte-t-elle la qualité du RAG ?
5. Comment le RAG réduit-il les hallucinations (sans les éliminer complètement) ?

### Pièges courants

- **"RAG = pas d'hallucinations"** : faux. Si le document récupéré contient une erreur, ou si la recherche récupère les mauvais documents, le modèle peut quand même produire des réponses incorrectes.
- **Confondre RAG et fine-tuning** : le RAG injecte de la connaissance au moment de la question ; le fine-tuning modifie le modèle lui-même (voir M1.7). Ils répondent à des besoins différents.
- **Négliger la qualité des données source** : garbage in, garbage out. Un RAG sur une documentation obsolète ou mal structurée donnera des réponses mauvaises.
- **Oublier les citations** : un bon produit RAG doit citer ses sources pour permettre à l'utilisateur de vérifier.

### Limites du concept

Le RAG ne fonctionne bien que si la base de connaissances est bien structurée, à jour, et de qualité. Il ne peut pas répondre à des questions qui n'ont pas de réponse dans les documents fournis, et ses performances se dégradent si les documents sont contradictoires.

## 3. Application Produit / Design

### Cas d'usage concrets

**Cas 1 : Chatbot de documentation produit**
Une entreprise SaaS intègre toute sa documentation (guides, FAQ, changelogs) dans une base vectorielle. Les utilisateurs posent des questions en langage naturel ("comment exporter mes données en CSV ?") et reçoivent une réponse précise avec un lien vers la source. Résultat typique : réduction de 40-60% des tickets support de niveau 1.

**Cas 2 : Assistant de recherche utilisateur**
Un product designer indexe 200 rapports d'entretiens utilisateurs. Il pose la question : "Quels sont les principaux points de friction lors de l'onboarding ?" Le système retrouve les verbatims pertinents et génère une synthèse en quelques secondes.

**Cas 3 : Moteur de recommandation de contenu personnalisé**
Une plateforme d'apprentissage utilise le RAG pour recommander des ressources : à partir du profil de l'apprenant et de sa question, elle retrouve les modules les plus pertinents dans son catalogue.

### Traduction IA → produit

| Feature classique | Feature RAG |
|---|---|
| Moteur de recherche par mots-clés | Recherche sémantique avec RAG |
| FAQ statique | Chatbot RAG sur la documentation |
| Résumés manuels de rapports | Synthèse automatique de verbatims |
| Recommandations par tags | Recommandations sémantiques |

### Exemple d'intégration produit réel

Perplexity AI est un moteur de recherche construit sur RAG : à chaque question, il effectue des recherches web en temps réel, récupère les sources pertinentes, et génère une réponse synthétique avec citations. C'est exactement le schéma RAG appliqué au web entier.

## 4. Pratique

### Exercice 1 — Concevoir l'architecture RAG d'un produit (20 min)

Pour un produit que tu connais (le tien, ou un concurrent), conçois une architecture RAG simple :
1. **Quelle base de connaissances ?** (documentation, FAQ, emails, articles, etc.)
2. **Quelle question type un utilisateur poserait-il ?**
3. **Quels documents devraient être récupérés pour y répondre ?**
4. **Quel serait le prompt système qui encadre la génération ?**
5. **Comment afficher les sources à l'utilisateur ?**

### Exercice 2 — Tester un produit RAG existant (15 min)

Utilise Perplexity.ai ou NotionAI (si tu l'as) et teste ses limites RAG :
- Pose une question très spécifique → observe si les sources sont pertinentes
- Pose une question hors scope → observe comment le système gère l'absence de réponse
- Pose une question sur un sujet récent → observe si le système reconnaît ses limites temporelles

### Mini-projet

**"Design d'un assistant RAG pour ton équipe"**

Conçois le cahier des charges d'un assistant RAG pour un besoin réel dans ton équipe (support, onboarding, knowledge base interne, etc.) :
1. Problème utilisateur ciblé
2. Sources de données à indexer
3. Interface utilisateur proposée (chatbot, barre de recherche, widget, etc.)
4. Stratégie pour gérer les questions sans réponse
5. Mécanisme de mise à jour des données
6. Comment tu mesurerais le succès

*Livrable : un document de 1 page utilisable comme brief pour une équipe tech.*

### Pour aller plus loin (profil technique)

Stack technique standard : **LlamaIndex** ou **LangChain** pour l'orchestration, **pgvector** ou **Pinecone** comme base vectorielle, **OpenAI text-embedding-3-small** ou **Cohere embed** pour les embeddings. Un tutoriel LlamaIndex en 1 heure suffit pour un premier POC.

Maintenir la distinction :

- **Implémentation** : embeddings, base vectorielle, chunking
