# M2.5 — Patterns d'interface pour l'IA : UI/UX des outputs génératifs

> Comme les patterns de design UI classiques (cards, modals, navigation tabs), il existe des patterns spécifiques à l'IA qui ont fait leurs preuves. Ce module est un catalogue opérationnel : pour chaque pattern, quand l'utiliser, comment l'implémenter, et les anti-patterns à éviter.

---

## 1. Théorie

### Définition

Un pattern d'interface IA est une solution de design récurrente et éprouvée pour des problèmes spécifiques aux outputs génératifs ou probabilistes : gérer l'incertitude, afficher des suggestions, permettre le contrôle utilisateur, ou communiquer la progression d'un système IA. Ces patterns forment le vocabulaire de design des produits IA.

### Modèles mentaux

**Modèle 1 : Les 3 familles de patterns IA**
- **Patterns de présentation** : comment montrer l'output IA (inline, panneau latéral, modal, streaming)
- **Patterns de contrôle** : comment permettre à l'utilisateur de guider, corriger, ou rejeter l'IA
- **Patterns de confiance** : comment communiquer la fiabilité de l'output (sources, niveau de confiance, explications)

**Modèle 2 : La règle de la friction proportionnelle**
La friction imposée à l'utilisateur doit être proportionnelle au risque de l'action IA. Une suggestion de correction d'orthographe (risque faible) → friction quasi-nulle. Un email généré prêt à envoyer (risque moyen) → confirmation explicite. Une décision de suppression automatique (risque élevé) → validation obligatoire + résumé de l'action.

### Schéma conceptuel — Les 8 patterns essentiels

```
PATTERN 1 : Inline Suggestion (complétion)
  Contexte : suggestion de texte, code, ou données dans le flux de travail
  Présentation : texte en gris clair, police différente
  Contrôle : Tab pour accepter, continuer à taper pour ignorer
  Exemples : GitHub Copilot, Gmail Smart Compose
  Anti-pattern : suggestion trop longue ou trop fréquente → distraction

PATTERN 2 : Side-by-side Comparison
  Contexte : plusieurs variantes IA à comparer (copy, design, code)
  Présentation : 2-3 options côte à côte ou en liste
  Contrôle : sélection + option de modifier
  Exemples : Notion AI "Améliore ce texte" → 3 variantes
  Anti-pattern : trop d'options (>4) → paralysie du choix

PATTERN 3 : Progressive Disclosure
  Contexte : output long ou complexe (résumé + détail)
  Présentation : résumé visible → "Voir plus" pour le détail
  Contrôle : expansion à la demande
  Exemples : Perplexity AI (réponse + sources)
  Anti-pattern : résumé trop court qui perd des informations critiques

PATTERN 4 : Confidence Indicator
  Contexte : output avec incertitude variable
  Présentation : badge, couleur, ou texte ("haute confiance", "à vérifier")
  Contrôle : invitation à vérifier si confiance basse
  Exemples : outils médicaux IA, moteurs de traduction
  Anti-pattern : afficher une confiance élevée par défaut (crée une sur-confiance)

PATTERN 5 : Guided Input
  Contexte : l'utilisateur doit fournir un bon contexte pour que l'IA réponde bien
  Présentation : formulaire structuré ou prompts de guidage
  Contrôle : l'utilisateur remplit le contexte, l'IA génère
  Exemples : Magic Design de Figma, Jasper AI templates
  Anti-pattern : champ texte libre sans guidance → outputs médiocres

PATTERN 6 : Streaming Output
  Contexte : réponse longue qui se génère progressivement (LLM)
  Présentation : texte qui apparaît token par token
  Contrôle : possibilité d'arrêter la génération
  Exemples : Claude, ChatGPT
  Anti-pattern : afficher un loader sans streaming → anxiété pour les réponses longues

PATTERN 7 : Edit-in-place
  Contexte : output IA qu'on veut modifier sans tout régénérer
  Présentation : output éditable directement (inline editing)
  Contrôle : édition directe + option de régénérer seulement la partie modifiée
  Exemples : Notion AI, Otter.ai transcriptions
  Anti-pattern : forcer la régénération complète pour une petite correction

PATTERN 8 : Source Attribution
  Contexte : output factuellement ancré dans des sources (RAG)
  Présentation : numéros de référence ou liens vers les sources
  Contrôle : clic sur la source pour vérifier
  Exemples : Perplexity AI, Bing Copilot
  Anti-pattern : afficher des sources non vérifiables ou inventer des sources
```

## 2. Compréhension

### 5 questions de vérification

1. Dans quelle situation le pattern "Inline Suggestion" est-il contre-indiqué ?
2. Comment choisir entre "Side-by-side Comparison" et "Guided Input" pour un même cas d'usage ?
3. Pourquoi le "Streaming Output" améliore-t-il la perception de performance même si le temps total est identique ?
4. Quand est-il approprié de NE PAS afficher les sources d'un output IA ?
5. Comment le pattern "Confidence Indicator" peut-il se retourner contre la confiance utilisateur si mal utilisé ?

### Pièges courants

- **Copier des patterns sans contexte** : un pattern qui fonctionne dans un contexte (code completion) peut être désastreux dans un autre (décision médicale).
- **Multiplier les patterns dans une même interface** : une interface avec 5 mécanismes de feedback IA différents est confuse.
- **Patterns "for show"** : afficher un indicateur de confiance fictif (toujours "haute confiance") détruit la confiance à long terme.
- **Négliger les états de chargement** : un loader IA sans feedback visuel crée de l'anxiété — le streaming ou une animation de "réflexion" aide.

### Limites du concept

Ce catalogue couvre les patterns les plus courants en 2025. Les patterns évoluent avec les capacités des modèles — ce qui était un pattern avancé en 2023 (streaming) est devenu standard. Se tenir à jour avec les patterns émergents est une pratique continue.

## 3. Application Produit / Design

### Cas d'usage concrets

**Cas 1 : Assistant de rédaction — combinaison de patterns**
Dans Notion AI, pour la feature "Améliorer l'écriture" : Guided Input (l'utilisateur choisit son objectif : "plus concis", "plus formel") → Side-by-side Comparison (3 variantes) → Edit-in-place (modification directe de la variante choisie) → Streaming Output pour chaque régénération.

**Cas 2 : Chatbot de documentation — confiance et attribution**
Dans un chatbot RAG : Streaming Output (réponse progressive) + Source Attribution (liens vers les docs) + Confidence Indicator (badge "basé sur 3 sources" vs "réponse générale") + Edit-in-place absent (il n'y a pas d'output à éditer, juste une réponse).

**Cas 3 : Moteur de recommandation — suggestions légères**
Dans Spotify, les suggestions "En ce moment en tendance" utilisent le pattern Progressive Disclosure : liste courte visible → "Voir plus" pour le catalogue complet. Pas de Confidence Indicator (inutile pour de la musique), pas de Source Attribution.

### Traduction IA → produit

| Besoin produit | Pattern recommandé |
|---|---|
| Suggérer sans interrompre | Inline Suggestion |
| Proposer plusieurs options | Side-by-side Comparison |
| Réponse longue à dévoiler | Progressive Disclosure |
| Output factuel à vérifier | Source Attribution |
| Utilisateur doit guider l'IA | Guided Input |
| Réponse qui prend du temps | Streaming Output |
| Output éditable par l'utilisateur | Edit-in-place |
| Output dont la fiabilité varie | Confidence Indicator |

## 4. Pratique

### Exercice 1 — Audit de patterns (20 min)

Choisis une application qui utilise de l'IA (Grammarly, Notion, GitHub Copilot, ChatGPT, Figma). Identifie :
- Quels patterns des 8 sont utilisés ?
- Pour chaque pattern : est-il bien implémenté ? Qu'améliorerais-tu ?
- Y a-t-il des patterns manquants qui amélioreraient l'expérience ?

### Exercice 2 — Choisir les bons patterns (20 min)

Pour la feature IA de ton mini-projet (M2.3), sélectionne les 2-3 patterns les plus pertinents et justifie :
1. Pourquoi ce pattern est-il adapté à ce contexte ?
2. Comment l'implémenterais-tu concrètement ?
3. Quel anti-pattern dois-tu éviter ?

### Mini-projet

**"Design system IA — 3 composants"**

Conçois 3 composants réutilisables dans Figma (ou en description détaillée) pour ton produit :
1. Un composant "Suggestion IA" (avec variantes : haute/basse confiance)
2. Un composant "Output IA" (avec actions : accepter, modifier, rejeter, régénérer)
3. Un composant "Source IA" (attribution avec lien)

Pour chaque composant : spécifications visuelles + états (nominal, chargement, erreur).

*Livrable : 3 composants Figma documentés ou spécifications détaillées utilisables par un développeur.*
