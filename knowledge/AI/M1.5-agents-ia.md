# M1.5 — Agents IA : des outils aux systèmes autonomes

> Un agent IA n'est pas juste un LLM plus puissant — c'est un changement de paradigme. Au lieu de répondre à une question, il peut prendre une séquence de décisions, utiliser des outils, et accomplir une tâche complexe en plusieurs étapes. Comprendre ce concept est essentiel pour concevoir les produits IA de la prochaine génération.

---

## 1. Théorie

### Définition

Un agent IA est un système qui utilise un LLM comme moteur de raisonnement pour accomplir des tâches en plusieurs étapes, en alternant entre des phases de réflexion et d'action. Contrairement à un LLM qui génère une réponse en une passe, un agent peut percevoir son environnement, décider quelle action exécuter, appeler des outils externes (APIs, recherche web, code), observer le résultat, et itérer jusqu'à l'objectif.

### Modèles mentaux

**Modèle 1 : La boucle agentic (Perception → Raisonnement → Action)**
Un agent opère en boucle :
- **Percevoir** : reçoit un objectif et l'état actuel (contexte, résultats précédents, mémoire)
- **Raisonner** : le LLM décide quelle action effectuer ensuite
- **Agir** : exécute l'action (appel d'outil, recherche, calcul, écriture de fichier)
- **Observer** : intègre le résultat dans son contexte
- **Répéter** jusqu'à la complétion de la tâche

**Modèle 2 : L'agent comme chef de projet avec des sous-traitants**
Le LLM central est le "cerveau décisionnel" qui coordonne. Les outils (recherche web, calculatrice, API, code executor) sont ses "sous-traitants spécialisés". Il ne fait pas tout lui-même — il délègue au bon outil au bon moment.

### Schéma conceptuel

```
Objectif / Tâche utilisateur
        ↓
  ┌─────────────────────────────────┐
  │         AGENT LOOP              │
  │                                 │
  │  LLM (raisonnement)             │
  │    ↓ "Je dois faire X"          │
  │  Sélection d'outil              │
  │    ↓                            │
  │  Exécution outil                │
  │  ├── Recherche web              │
  │  ├── Calcul / code              │
  │  ├── Appel API externe          │
  │  ├── Lecture/écriture fichiers  │
  │  └── Appel d'un autre agent     │
  │    ↓                            │
  │  Résultat → Contexte mis à jour │
  │    ↓                            │
  │  Tâche terminée ? Non → boucle  │
  └─────────────────────────────────┘
        ↓ Oui
  Résultat final
```

### Analogie

"Un LLM sans agent, c'est un expert en salle de réunion qui répond aux questions. Un agent, c'est ce même expert qui peut aussi appeler des collègues, faire des recherches sur son ordinateur, rédiger des documents, et envoyer des emails — le tout de façon autonome pour accomplir une mission."

## 2. Compréhension

### 5 questions de vérification

1. Quelle est la différence fondamentale entre un LLM "standalone" et un agent IA ?
2. Pourquoi un agent peut-il se retrouver dans une boucle infinie, et comment l'éviter ?
3. Qu'est-ce qu'un "tool call" et comment il s'intègre dans la boucle agentic ?
4. Dans quel cas préférer un agent à un simple LLM avec RAG ?
5. Qu'est-ce qu'un système multi-agents et quel problème résout-il ?

### Pièges courants

- **"Plus d'autonomie = toujours mieux"** : les agents sont puissants mais risqués. Plus ils agissent de façon autonome sur des systèmes réels (emails, fichiers, APIs), plus les erreurs ont des conséquences. La supervision humaine reste essentielle.
- **Confondre agent et chatbot** : un chatbot répond à des questions. Un agent accomplit des tâches. La différence n'est pas cosmétique — elle change l'architecture et les risques.
- **Négliger les boucles d'erreur** : sans mécanisme d'arrêt explicite, un agent peut boucler indéfiniment en essayant des actions qui échouent.
- **Sur-estimer la fiabilité** : les agents héritent des hallucinations des LLM. Une mauvaise décision du LLM peut déclencher une cascade d'actions incorrectes.

### Limites du concept

Les agents actuels sont puissants mais peu fiables sur des tâches nécessitant une précision absolue ou un raisonnement très long. Ils excellent sur des tâches à étapes bien définies avec des outils clairs. Les systèmes multi-agents augmentent la puissance mais aussi la complexité de débogage.

## 3. Application Produit / Design

### Cas d'usage concrets

**Cas 1 : Agent de recherche concurrentielle**
Un PM décrit sa demande : "Fais un benchmark de nos 5 concurrents sur la feature X." L'agent cherche sur le web, lit les pages de landing, extrait les informations pertinentes, structure un tableau comparatif, et génère un résumé en 5 minutes. Ce qui prenait 3 heures se fait en autonomie.

**Cas 2 : Agent de triage et réponse aux issues GitHub**
Dans une équipe technique, un agent surveille les nouvelles issues, les classe par type (bug / feature request / question), génère une réponse initiale, et assigne au bon membre de l'équipe — en laissant un humain valider avant toute action irréversible.

**Cas 3 : Agent de qualification de leads**
Un agent reçoit un nouveau lead (email, formulaire), consulte le CRM, vérifie la taille de l'entreprise sur LinkedIn, génère un score de qualification, et rédige un email de suivi personnalisé — tout ça sans intervention humaine jusqu'à l'envoi.

### Traduction IA → produit

| Paradigme LLM simple | Paradigme agent |
|---|---|
| L'utilisateur pose une question, l'IA répond | L'utilisateur donne un objectif, l'IA accomplit |
| Interaction unique | Boucle de plusieurs actions |
| Pas d'accès aux outils | Outils + APIs + bases de données |
| Supervision à chaque étape | Supervision optionnelle (humain-in-the-loop) |
| Output : texte | Output : action réalisée dans le monde |

### Exemple d'intégration produit réel

Claude Projects avec des outils activés (recherche web, exécution de code) est un agent. Cursor (IDE avec IA) est un agent : il lit des fichiers, modifie du code, exécute des tests, observe les erreurs, et itère — tout ça en boucle, de façon quasi-autonome. Devin d'Cognition pousse encore plus loin : un agent qui peut mener un projet de développement complet.

## 4. Pratique

### Exercice 1 — Décomposer une tâche en boucle agentic (20 min)

Prends cette tâche : *"Prépare un rapport sur les tendances UX en 2025."*

Décris comment un agent accomplirait cette tâche étape par étape :
1. Liste les étapes (recherche, extraction, synthèse, formatage)
2. Pour chaque étape : quel outil utilise l'agent ? Quel est le résultat attendu ?
3. Quels points de décision l'agent doit-il gérer (ex : source non disponible, résultat ambigu) ?
4. Où placerais-tu un "checkpoint humain" pour valider avant de continuer ?

### Exercice 2 — Analyser un agent existant (20 min)

Utilise Claude avec des outils activés (ou un agent no-code comme Make/n8n si disponible). Donne-lui une tâche en plusieurs étapes et observe :
- Combien d'étapes effectue-t-il ?
- Quels outils appelle-t-il ?
- Où fait-il des erreurs ou des approximations ?
- Où aurait-il besoin de supervision humaine ?

### Mini-projet

**"Design d'un agent pour ton équipe"**

Identifie une tâche répétitive dans ton équipe (rédaction de briefs, analyse de données, triage de demandes) qui pourrait être partiellement automatisée par un agent. Conçois :
1. L'objectif de l'agent (input → output)
2. Les outils dont il a besoin
3. Les étapes de la boucle agentic
4. Les points de validation humaine (humain-in-the-loop)
5. Les risques et comment les mitiger
6. Le critère de succès

*Livrable : un flowchart de l'agent avec les checkpoints humains explicitement marqués.*

## 8. Contraintes pédagogiques

Ne pas passer directement aux outils d'agents sans comprendre la boucle agentic conceptuellement. Le risque est de construire un "agent" sans comprendre ses points de défaillance.

Distinction à maintenir :
- **Concept** : boucle perception → raisonnement → action
- **Implémentation** : tool calls, function calling, frameworks (LangGraph, CrewAI)
- **Usage produit** : automatisation de tâches, assistants autonomes, qualification de leads
