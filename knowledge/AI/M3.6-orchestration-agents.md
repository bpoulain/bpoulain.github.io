# M3.6 — Orchestration d'agents pour des processus métier complexes

> Un agent unique peut accomplir une tâche en plusieurs étapes. Un système multi-agents peut gérer des processus entiers : qualifier un lead, écrire un brief, et planifier un projet — en parallèle, avec des spécialisations complémentaires. Ce module explore les architectures multi-agents appliquées aux processus métier.

---

## 1. Théorie

### Définition

L'orchestration d'agents est la coordination de plusieurs agents IA spécialisés qui travaillent ensemble pour accomplir un processus complexe. Chaque agent a un rôle précis et des outils dédiés. Un orchestrateur (humain ou IA) coordonne leurs actions, gère les dépendances, et agrège les résultats.

### Modèles mentaux

**Modèle 1 : L'équipe projet IA**
Un système multi-agents fonctionne comme une équipe : il y a un manager (l'orchestrateur), des spécialistes (les agents), et des outils partagés. Le manager décompose la tâche, distribue le travail, supervise la progression, et consolide les résultats. La clé : chaque spécialiste est excellent dans son domaine, et mauvais dans celui des autres.

**Modèle 2 : Les 3 topologies d'orchestration**
```
SÉQUENTIEL           PARALLÈLE              HIÉRARCHIQUE
Agent A              Agent A   Agent B       Orchestrateur
  ↓                     ↓         ↓              ↓      ↓
Agent B              Agent C   Agent D      Équipe A  Équipe B
  ↓                     ↓         ↓            ↓↓       ↓↓
Agent C              Aggrégation            Agents  Agents

Cas d'usage :        Cas d'usage :          Cas d'usage :
Pipeline linéaire    Tâches indépendantes   Processus complexes
(A dépend de B)      en parallèle           multi-niveaux
```

### Schéma conceptuel

```
EXEMPLE : Système multi-agents pour le cycle de recrutement

ORCHESTRATEUR
  Reçoit : "Recruter un UX Designer senior"
  Décompose et distribue...
                │
    ┌───────────┼───────────┐
    ↓           ↓           ↓
AGENT           AGENT       AGENT
"Sourcing"     "Scoring"   "Scheduling"
    │               │           │
Scrape LinkedIn  Analyse CVs  Coordonne
Génère outreach  Score 1-10   calendriers
Posts job board  Recommande   Envoie invites
    │               │           │
    └───────────────┴───────────┘
                    ↓
             ORCHESTRATEUR
             Consolide les données
             Génère un rapport
             Notifie le recruteur
                    ↓
         VALIDATION HUMAINE OBLIGATOIRE
         avant toute décision de contact
```

### Analogie

"Un système multi-agents, c'est comme un cabinet de conseil : il y a un partner (l'orchestrateur) qui comprend le client et distribue le travail, des consultants spécialisés (les agents) qui font chacun leur partie, et des outils partagés (CRM, bases de données). Le client (l'utilisateur) interagit principalement avec le partner, pas avec chaque consultant."

## 2. Compréhension

### 5 questions de vérification

1. Pourquoi une topologie parallèle n'est-elle pas toujours meilleure qu'une topologie séquentielle, même quand les tâches sont indépendantes ?
2. Comment gérer les erreurs dans un pipeline séquentiel : si l'agent B échoue, que fait l'orchestrateur ?
3. Qu'est-ce que le "context sharing" entre agents et pourquoi c'est un défi technique ?
4. Comment décider du niveau de supervision humaine dans un système multi-agents (à quelle fréquence demander validation) ?
5. Quels sont les risques de sécurité spécifiques aux systèmes multi-agents qui interagissent avec des systèmes externes ?

### Pièges courants

- **Trop d'agents** : chaque agent ajouté augmente la complexité, la latence, et le coût. Commencer avec le minimum viable (souvent 1-2 agents bien définis).
- **Agents trop généralistes** : un agent qui "fait tout" n'est pas meilleur qu'un LLM seul. La valeur des multi-agents vient de la spécialisation.
- **Boucles sans fin** : sans mécanisme d'arrêt explicite, les agents peuvent se transmettre des tâches indéfiniment. Toujours implémenter un max_iterations.
- **Confiance aveugle entre agents** : un agent peut transmettre des données incorrectes à un autre qui les accepte sans validation. Les guardrails s'appliquent aussi aux inter-communications.
- **Ignorer les coûts** : chaque appel LLM a un coût. Un pipeline multi-agents peut consommer 10-50x plus de tokens qu'un LLM simple.

## 3. Application Produit / Design

### Cas d'usage concrets

**Cas 1 : Pipeline de création de contenu marketing**
Agent 1 (Recherche) : recherche les tendances et la concurrence sur un sujet → Agent 2 (Brief) : génère un brief de contenu structuré → Agent 3 (Rédaction) : produit un premier draft → Agent 4 (Révision) : optimise pour le SEO et le ton de marque → Output : article prêt pour validation humaine.
Gain estimé : 6h de travail réduit à 20 min + validation de 30 min.

**Cas 2 : Agent de qualification et routing des leads**
Agent 1 (Enrichissement) : enrichit les données du lead (LinkedIn, site web, news) → Agent 2 (Qualification) : score le lead selon les critères ICP → Agent 3 (Routing) : assigne au bon commercial avec un brief personnalisé → Notification Slack + tâche CRM créée.
Supervision humaine : le commercial revoit le scoring avant toute action.

**Cas 3 : Assistant de veille concurrentielle continue**
Agent 1 (Monitoring) : surveille les sites concurrents, Product Hunt, Twitter pour les nouvelles features → Agent 2 (Analyse) : compare avec la roadmap interne → Agent 3 (Synthèse) : génère un rapport hebdomadaire avec recommandations → Distribution automatique à l'équipe produit.

### Traduction IA → architecture multi-agents

| Caractéristique du processus | Architecture recommandée |
|---|---|
| Tâches séquentielles avec dépendances | Pipeline séquentiel |
| Tâches indépendantes | Parallèle (gain de temps) |
| Processus de décision complexe | Hiérarchique avec orchestrateur |
| Processus itératif (aller-retour) | Boucle agentic avec checkpoints |
| Volume élevé et standardisé | Pipeline entièrement automatisé |
| Décisions à fort impact | Obligatoirement avec checkpoint humain |

## 4. Pratique

### Exercice 1 — Décomposer un processus en agents (30 min)

Prends un processus complexe de ton équipe (recrutement, onboarding, création de contenu, analyse concurrentielle). Décompose-le en agents :
1. Quels agents sont nécessaires ? (nom + rôle)
2. Quelle topologie ? (séquentiel / parallèle / hiérarchique)
3. Quels outils chaque agent nécessite-t-il ?
4. Quels sont les checkpoints humains obligatoires ?
5. Quels sont les points de défaillance possibles ?

### Exercice 2 — Construire un mini pipeline multi-agents (30 min)

Avec n8n (open source) ou Make, construis un pipeline de 2 agents séquentiels :
1. Agent 1 : reçoit une URL d'article → extrait le contenu → résume en 5 points clés
2. Agent 2 : reçoit le résumé → génère 3 questions de discussion → formate pour Slack
3. Déclencher sur un trigger (nouveau lien dans un canal Slack, par exemple)

*Critère de réussite : le pipeline tourne sans erreur sur 3 articles test.*

### Mini-projet

**"Architecture d'un système multi-agents pour mon équipe"**

Conçois l'architecture complète d'un système multi-agents pour automatiser un processus important de ton équipe :
1. Description du processus cible et de ses enjeux
2. Schéma de l'architecture (agents + connexions + outils + checkpoints)
3. Estimation ROI (temps économisé / coût de développement)
4. Plan de déploiement progressif (commencer par quoi ?)
5. Indicateurs de succès et critères d'arrêt

*Livrable : document d'architecture prêt pour présentation à l'équipe tech.*
