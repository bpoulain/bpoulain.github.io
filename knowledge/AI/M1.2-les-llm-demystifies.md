# M1.2 — Les LLM démystifiés : comment un modèle de langage pense

> Le module le plus important du système. Comprendre comment un LLM fonctionne réellement change la façon dont tu conçois des produits, tu écris des prompts, et tu débogues des comportements inattendus. Ce n'est pas de la magie — c'est de la statistique à grande échelle.

---

## 1. Théorie

### Définition

Un Large Language Model (LLM) est un modèle statistique entraîné sur des centaines de milliards de mots pour prédire, token par token, la suite la plus probable d'un texte donné. Il ne "comprend" pas — il associe des patterns dans un espace vectoriel de très haute dimension. Sa puissance vient de la quantité de patterns mémorisés, pas d'une forme de conscience ou de raisonnement.

### Modèles mentaux

**Modèle 1 : La prédiction de token suivant**
Un LLM est fondamentalement un moteur de prédiction. À chaque étape, il reçoit tout le texte précédent et calcule la probabilité de chaque token possible comme suite. Il choisit ensuite un token (souvent le plus probable, parfois avec une part d'aléatoire contrôlée par la "température"). Répété des centaines de fois, ce processus génère un texte cohérent. *Conséquence design : le modèle n'a pas d'intention — il a des probabilités.*

**Modèle 2 : L'espace vectoriel comme carte du sens**
Chaque mot, phrase ou concept est représenté comme un point dans un espace à des milliers de dimensions. Les concepts proches sémantiquement sont proches dans cet espace. "Roi" moins "Homme" plus "Femme" ≈ "Reine". *Conséquence design : le modèle peut raisonner par analogie et généraliser à des situations qu'il n'a pas vues explicitement.*

### Schéma conceptuel

```
Entrée utilisateur (prompt)
        ↓
  Tokenisation
  "Bonjour" → [48926] | "je" → [492] | "suis" → [14564] ...
        ↓
  Embedding (chaque token → vecteur)
        ↓
  Mécanisme d'attention (Transformer)
  "quel token est pertinent pour quel autre token ?"
        ↓
  Calcul des probabilités sur le vocabulaire (~100 000 tokens)
        ↓
  Sélection du prochain token (déterministe ou probabiliste)
        ↓
  Répéter jusqu'au token de fin
        ↓
  Sortie (texte généré)
```

### Analogie

"Un LLM, c'est comme un étudiant qui aurait lu toute la bibliothèque d'internet, sans jamais l'avoir vécue. Il peut écrire sur la plupart des sujets avec cohérence, mais il n'a aucune expérience personnelle et peut confabuler avec la même assurance qu'un expert."

## 2. Compréhension

### 5 questions de vérification

1. Pourquoi un LLM peut-il répondre avec confiance à une question dont la réponse est fausse ?
2. Qu'est-ce qu'un token ? Combien de tokens correspondent environ à 1000 mots ?
3. Que signifie "température élevée" dans la génération d'un LLM, et quel impact sur le résultat ?
4. Pourquoi le même prompt peut donner des réponses différentes d'une session à l'autre ?
5. Quelle est la limite fondamentale d'un LLM sans mémoire externe ou accès au web ?

### Pièges courants

- **Les hallucinations ne sont pas des bugs** : elles sont une conséquence directe du mécanisme de prédiction. Le modèle génère toujours ce qui semble le plus probable, même si c'est factuellement faux.
- **"Plus grand = meilleur" n'est pas toujours vrai** : un modèle plus petit et mieux affiné sur un domaine peut surpasser un grand modèle généraliste sur ce domaine spécifique.
- **Le contexte est tout** : les LLM n'ont pas de mémoire entre les sessions. Chaque conversation repart de zéro — sauf si tu injectes explicitement le contexte.

### Limites du concept

Ce module couvre le fonctionnement général mais ne détaille pas l'architecture transformer en profondeur, ni le processus d'entraînement (RLHF, fine-tuning). Ces aspects sont abordés en M1.7.

## 3. Application Produit / Design

### Cas d'usage concrets

**Cas 1 : Assistant de rédaction dans un CMS**
Un LLM suggère des titres, des reformulations et des métadescriptions à partir du contenu d'un article. Il tire parti de sa connaissance des patterns d'écriture web pour proposer des variantes pertinentes — mais l'auteur reste le validateur final, car le modèle peut inventer des statistiques ou déformer des faits.

**Cas 2 : Chatbot de support produit**
Un LLM répond aux questions des utilisateurs sur un produit SaaS. Sans contexte injecté (documentation, historique), il peut halluciner des fonctionnalités inexistantes. Avec du RAG (voir M1.4), il répond à partir de la vraie documentation — la différence est radicale.

**Cas 3 : Analyse de verbatims utilisateurs**
Un LLM lit 500 verbatims d'une enquête et produit une synthèse thématique en 5 minutes. Tâche qui prenait 2 jours à un analyste. Le risque : le modèle peut sur-représenter certains thèmes ou en fusionner qui méritent d'être distincts.

### Traduction IA → produit

| Comportement LLM | Implication design |
|---|---|
| Génération probabiliste | Prévoir la validation humaine pour les outputs critiques |
| Contexte limité (context window) | Concevoir des interfaces qui gèrent les longues sessions |
| Pas de mémoire native | Implémenter une mémoire explicite si le produit le requiert |
| Hallucinations possibles | Ne jamais afficher un output LLM brut sans mécanisme de confiance |
| Force en génération, faiblesse en calcul | Ne pas utiliser un LLM pour des calculs précis |

### Exemple d'intégration produit réel

GitHub Copilot utilise un LLM (GPT-4 / Codex) qui a été entraîné sur des milliards de lignes de code. Quand tu tapes du code, il prédit la suite token par token — exactement comme il prédit du texte, sauf que le "texte" est du Python ou du JavaScript. La magie, c'est la quantité de patterns de code mémorisés, pas une compréhension de la logique de ton programme.

## 4. Pratique

### Exercice 1 — Tester les limites (20 min)

Avec Claude ou ChatGPT, teste les scénarios suivants et note tes observations :

1. Pose une question factuelle précise (ex : "Quel est le PIB de la France en 2024 ?") → Est-ce précis ? Vérifiable ?
2. Pose la même question deux fois dans deux sessions séparées → Les réponses sont-elles identiques ?
3. Demande-lui d'inventer une biographie d'une personne fictive → Qu'est-ce que ça révèle sur son mécanisme ?
4. Donne-lui un contexte faux délibérément ("tu es un expert en pâtes culinaires appelé Marco") → Comment s'adapte-t-il ?

*Critère de réussite : tu peux expliquer chaque comportement observé avec le modèle mental de la prédiction de token.*

### Exercice 2 — Analyser un output pour comprendre le mécanisme (15 min)

Prends un output d'un LLM que tu trouves surprenant ou imparfait. Analyse-le :
- Pourquoi ce token/mot a-t-il semblé le plus probable à cet endroit ?
- Quels patterns dans les données d'entraînement peuvent expliquer ce choix ?
- Quelle modification du prompt aurait orienté le modèle vers un meilleur output ?

### Mini-projet

**"Documentation des comportements LLM"**

Pour un produit qui intègre ou pourrait intégrer un LLM :
1. Liste 5 comportements attendus du LLM dans ce contexte
2. Pour chacun, documente le risque d'hallucination ou d'échec
3. Propose une stratégie de mitigation pour chaque risque (validation humaine, guardrails, RAG, etc.)

*Livrable : un tableau risques/mitigations prêt pour une revue de design.*

## 8. Contraintes pédagogiques

Ce module doit être lu **avant** d'utiliser des outils LLM de façon intensive. La compréhension du mécanisme de prédiction transforme l'expérience d'un utilisateur passif en un utilisateur qui comprend pourquoi ça marche et pourquoi ça échoue.

Distinction à maintenir tout au long du module :
- **Concept** : prédiction probabiliste de tokens
- **Implémentation** : architecture transformer + RLHF
- **Usage produit** : génération de texte, analyse, classification
