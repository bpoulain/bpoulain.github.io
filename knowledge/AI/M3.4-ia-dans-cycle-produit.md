# M3.4 — Intégrer l'IA dans le cycle produit : discovery, delivery, rétro

> L'IA ne s'intègre pas uniquement dans le produit qu'on livre — elle peut transformer le processus par lequel on le livre. Ce module cartographie les opportunités IA à chaque phase du cycle produit, de la discovery à la rétrospective.

---

## 1. Théorie

### Définition

L'intégration de l'IA dans le cycle produit désigne l'utilisation d'outils et de techniques IA pour améliorer l'efficacité des équipes produit dans leur processus de travail : recherche utilisateur, priorisation, rédaction, revue, analyse. L'objectif n'est pas de remplacer les décisions humaines, mais d'accélérer les tâches à faible valeur ajoutée pour libérer du temps pour les activités stratégiques.

### Modèles mentaux

**Modèle 1 : L'IA comme couche d'amplification**
À chaque phase du cycle produit, l'IA peut amplifier l'efficacité de l'équipe en prenant en charge les tâches répétitives ou de traitement d'information :
- Discovery : synthèse de données, extraction d'insights, génération d'hypothèses
- Design : génération de variantes, feedback sur les prototypes, rédaction
- Delivery : rédaction de tickets, documentation, revue de code
- Post-launch : analyse des métriques, synthèse des retours, reporting

**Modèle 2 : Le gradient de délégation**
Chaque tâche peut être déléguée à différents niveaux :
- **Niveau 0** : entièrement humaine (décisions stratégiques, priorisation finale)
- **Niveau 1** : IA prépare, humain décide (brief d'entretien, premier jet de ticket)
- **Niveau 2** : IA fait, humain valide (résumé de réunion, release notes)
- **Niveau 3** : IA fait, humain surveille (triage, monitoring, alertes)

### Schéma conceptuel

```
CYCLE PRODUIT — CARTE D'INTÉGRATION IA

DISCOVERY
  Entretiens utilisateurs
  ├── Avant : LLM génère le guide d'entretien + hypothèses    [Niveau 1]
  ├── Pendant : transcription automatique (Otter/Whisper)     [Niveau 3]
  └── Après : LLM synthétise les thèmes + verbatims clés      [Niveau 2]
  
  Benchmark concurrentiel
  └── Agent recherche + synthèse automatique                  [Niveau 2]
  
  Analyse de données quantitatives
  └── LLM interprète les graphes + identifie les anomalies    [Niveau 1]

DESIGN
  Rédaction du brief
  └── LLM génère le premier draft à partir des insights       [Niveau 1]
  
  Copy et microcopy
  └── LLM génère des variantes selon le ton de marque         [Niveau 1]
  
  Documentation des décisions
  └── LLM structure les notes de réunion en ADR               [Niveau 2]

DELIVERY
  Rédaction des user stories
  └── LLM transforme les notes en tickets formattés           [Niveau 2]
  
  Release notes
  └── LLM génère depuis le git log ou les tickets             [Niveau 2]
  
  Tests
  └── LLM génère des cas de test depuis les user stories      [Niveau 1]

POST-LAUNCH
  Analyse des métriques
  └── LLM interprète les chiffres et suggère des hypothèses   [Niveau 1]
  
  Triage des retours utilisateurs
  └── Classification automatique + résumé hebdomadaire        [Niveau 3]
  
  Rapport de sprint / mensuel
  └── LLM génère le premier draft du rapport                  [Niveau 2]
```

### Analogie

"Intégrer l'IA dans son cycle produit, c'est comme avoir un très bon assistant junior dans chaque phase : il prépare les briefs, prend les notes, fait les premiers drafts, résume les longs documents. Il ne remplace pas le Product Manager ou le Designer — il absorbe la charge cognitive répétitive pour qu'ils se concentrent sur la réflexion stratégique."

## 2. Compréhension

### 5 questions de vérification

1. Pourquoi la priorisation finale du backlog ne devrait-elle pas être déléguée à une IA ?
2. Comment garantir que l'IA ne biaise pas les insights discovery en sur-représentant certains thèmes ?
3. À quelles conditions les release notes générées par IA peuvent-elles être envoyées sans relecture humaine ?
4. Comment intégrer des outils IA dans un cycle produit sans perturber les workflows existants de l'équipe ?
5. Quel est le risque de dépendance si l'équipe produit délègue trop à l'IA ?

### Pièges courants

- **Résumé d'entretien sans vérification** : un LLM peut rater des nuances cruciales dans un entretien utilisateur. La vérification par le researcher reste indispensable.
- **Tickets générés sans contexte suffisant** : des user stories générées par IA sans contexte produit précis seront génériques et peu utiles.
- **Over-documentation** : l'IA rend la documentation si facile que les équipes peuvent créer trop de documents inutilisés.
- **Dépendance aux outils** : si toutes les réunions sont résumées par IA, les participants n'écoutent plus vraiment — la prise de notes manuelle force l'engagement.

### Limites du concept

Ce module présente un idéal d'intégration. En pratique, l'adoption se fait progressivement, et chaque équipe doit identifier les phases où l'IA apporte le plus de valeur **dans son contexte spécifique**. Il n'existe pas de recette universelle.

## 3. Application Produit / Design

### Cas d'usage concrets

**Cas 1 : Discovery accélérée — équipe de 2 personnes**
Une startup avec un petit product team conduit 10 entretiens utilisateurs en 1 semaine. Sans IA : synthèse = 2 jours. Avec Otter.ai (transcription) + Claude (synthèse thématique) : synthèse = 3 heures. Le PM valide et enrichit. Le temps économisé va dans une analyse supplémentaire de données analytics.

**Cas 2 : Delivery structurée — feature de recherche**
À chaque fin de sprint, un LLM reçoit les notes du sprint review et génère automatiquement : le résumé du sprint, les décisions prises, les risques identifiés, et les questions ouvertes pour le prochain sprint. Le PM valide en 15 min au lieu de 45.

**Cas 3 : Post-launch — monitoring continu**
Un workflow automatique (Make + Claude) collecte chaque nuit les nouveaux avis App Store + les tickets support + les mentions Twitter. Chaque matin, l'équipe reçoit un digest de 5 points clés + un score de sentiment. Tâche qui prenait 1h/jour → 5 min de lecture.

### Traduction IA → processus produit

| Phase | Tâche cible | Outil / Approche | Économie estimée |
|---|---|---|---|
| Discovery | Synthèse entretiens | Claude + Otter | 70% du temps de synthèse |
| Discovery | Benchmark concurrentiel | Agent recherche | 60% du temps de benchmark |
| Design | Variantes de copy | LLM | 50% du temps de rédaction |
| Delivery | Rédaction tickets | LLM depuis notes | 40% du temps de rédaction |
| Delivery | Release notes | LLM depuis git/tickets | 80% du temps de rédaction |
| Post-launch | Synthèse retours | Workflow automatique | 90% de la collecte/tri |

## 4. Pratique

### Exercice 1 — Cartographie personnelle (20 min)

Cartographie ton propre cycle de travail de la semaine dernière :
1. Liste les 10 tâches que tu as effectuées (par phase)
2. Pour chacune : durée réelle, niveau de délégation possible (0-3)
3. Identifie les 3 tâches à plus fort potentiel d'accélération IA
4. Calcule le temps total récupérable par semaine

### Exercice 2 — Tester un outil IA dans ton flux de travail réel (20 min)

Prends la tâche à plus fort potentiel identifiée. Utilise un LLM pour l'effectuer cette semaine :
- Documente le prompt utilisé
- Compare l'output IA avec ce que tu aurais produit seul
- Note le gain de temps réel (pas estimé)
- Note les ajustements nécessaires

### Mini-projet

**"Plan d'intégration IA pour mon équipe"**

Produis un plan d'intégration IA en 1 page pour ton équipe :
1. Phases ciblées (quelles phases du cycle ?)
2. Outils recommandés (avec justification)
3. Modifications de processus nécessaires
4. Plan de montée en compétences
5. Indicateurs de succès (comment savoir si ça marche ?)
6. Risques et garde-fous

*Livrable : document prêt à présenter lors d'une réunion d'équipe.*
