# M2.4 — Évaluer la qualité d'une feature IA : métriques produit

> "Est-ce que notre feature IA fonctionne bien ?" est une question à laquelle il est facile de répondre avec les mauvaises métriques. Ce module donne le framework pour mesurer la valeur réelle d'une feature IA — du point de vue utilisateur ET du point de vue produit.

---

## 1. Théorie

### Définition

Évaluer une feature IA requiert deux niveaux de métriques distincts : les **métriques techniques** (performances du modèle : précision, rappel, latence) et les **métriques produit** (valeur utilisateur : adoption, satisfaction, impact sur les KPIs produit). Les premières sans les secondes donnent une image incomplète ; les secondes sans les premières masquent des problèmes de fiabilité.

### Modèles mentaux

**Modèle 1 : La pyramide de l'évaluation IA**
```
      Impact business
         (revenus, rétention, NPS)
             ↑
      Valeur utilisateur
         (adoption, satisfaction, efficacité)
             ↑
      Qualité de l'output
         (précision perçue, cohérence, pertinence)
             ↑
      Performance technique
         (précision modèle, latence, disponibilité)
```
Une feature peut scorer très bien en bas de la pyramide (bon modèle) et mal en haut (pas de valeur perçue). L'inverse est aussi possible (valeur perçue élevée malgré un modèle imparfait).

**Modèle 2 : Les 4 types de métriques produit IA**
1. **Adoption** : les utilisateurs utilisent-ils la feature ? (taux d'activation, fréquence)
2. **Qualité perçue** : les utilisateurs trouvent-ils les outputs utiles ? (feedback explicite + implicite)
3. **Efficacité** : la feature accélère-t-elle la tâche cible ? (temps réduit, étapes évitées)
4. **Confiance** : les utilisateurs font-ils confiance au système dans le temps ? (taux de correction, override)

### Schéma conceptuel

```
FRAMEWORK D'ÉVALUATION D'UNE FEATURE IA

┌─────────────────────────────────────────────────────────┐
│  NIVEAU 1 — Technique                                   │
│  • Latence (p50, p95)                                   │
│  • Taux d'erreur technique (timeouts, failures)         │

│  • Taux de régénération (demande de nouvel essai)       │
│  • Feedback explicite (pouce haut/bas, étoiles)         │
└──────────────────────┬──────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────────┐
│  NIVEAU 3 — Valeur utilisateur                          │
│  • Adoption (% users qui utilisent la feature)          │
│  • Rétention (utilisent-ils toujours dans 30 jours ?)   │

```

### Analogie

"Évaluer une feature IA, c'est comme évaluer un traducteur. Il peut produire des traductions techniquement correctes (niveau 1) qui ne sont pas utilisables par les clients parce que le ton est froid (niveau 2), que les traducteurs humains ne font pas confiance à ses suggestions (niveau 3), et que l'entreprise ne voit pas de réduction de coût (niveau 4). Les 4 niveaux doivent être évalués."

## 2. Compréhension

### 5 questions de vérification

1. Pourquoi un taux d'acceptation sans modification élevé peut-il être un mauvais signal ?
2. Comment distinguer un utilisateur qui "fait confiance" à l'IA d'un utilisateur qui "ne vérifie plus" (et pourquoi c'est important) ?
3. Que révèle un taux de correction élevé sur la qualité d'une feature IA ?
4. Comment construire un benchmark de qualité humain pour évaluer les outputs d'un LLM ?
5. Dans quelle situation les métriques d'adoption peuvent-elles masquer un problème de qualité ?

### Pièges courants

- **Utiliser uniquement les métriques techniques** : un modèle avec une précision de 90% peut être perçu comme mauvais si les 10% d'erreurs affectent les cas les plus importants.
- **Confondre adoption et valeur** : une feature peut être adoptée par habitude ou par manque d'alternative, sans créer de vraie valeur.
- **Négliger le feedback implicite** : les signaux comportementaux (taux de correction, taux de régénération) sont souvent plus honnêtes que les sondages.
- **Évaluer sans baseline** : impossible de mesurer l'amélioration sans savoir comment la tâche était accomplie avant.

### Limites du concept

Ce module couvre l'évaluation d'une feature existante. Il ne couvre pas les tests A/B (inclus dans un cours de product management général) ni les mécanismes de feedback pour améliorer le modèle dans le temps (voir M2.6).

## 3. Application Produit / Design

### Cas d'usage concrets

**Cas 1 : Assistant de rédaction d'emails**
Métriques clés : taux d'utilisation de la suggestion (adoption), % d'emails envoyés sans modification (qualité), temps moyen de rédaction avec vs sans l'assistant (efficacité), score NPS de la feature (satisfaction). Signal d'alarme : si 80% des emails sont envoyés sans modification mais que le taux de réponse chute, le modèle optimise peut-être la vitesse au détriment de la qualité.

**Cas 2 : Feature de résumé automatique de documents**
Baseline : temps moyen pour lire et synthétiser un document (avant). Après : temps de lecture du résumé + validation. Taux de correction du résumé (qualité perçue). Taux de réutilisation dans les 2 semaines (rétention). Verbatims utilisateurs (CSAT).

**Cas 3 : Recommandation de contenu dans une app**
Taux de clic sur les recommandations (adoption), taux de complétion du contenu recommandé vs contenu auto-sélectionné (qualité), impact sur l'engagement hebdomadaire (impact business).

### Traduction IA → produit

| Ce qu'on mesure | Ce que ça révèle |
|---|---|
| Taux d'acceptation sans modification | Qualité et pertinence de l'output |
| Taux de correction | Où le modèle échoue systématiquement |
| Taux de régénération | L'output est trop mauvais pour être corrigé |
| Taux d'abandon après IA | La feature crée plus de friction qu'elle n'en résout |
| Feedback explicite + implicite | Corrélation : si le feedback explicite est bon mais le feedback implicite mauvais, les utilisateurs ne disent pas la vérité |

### Exemple d'intégration produit réel

GitHub Copilot mesure son succès principalement par le "acceptance rate" (% de suggestions acceptées) et la rétention des utilisateurs qui l'ont activé. Ces deux métriques sont plus révélatrices que la précision technique du modèle — parce qu'un développeur qui revient chaque jour sur Copilot vote avec son comportement.

## 4. Pratique

### Exercice 1 — Définir les métriques d'une feature (20 min)

Prends la feature IA prototypée en M2.3. Définis pour elle :
- 2 métriques techniques
- 3 métriques de qualité d'output
- 2 métriques de valeur utilisateur
- 1 métrique d'impact business

Pour chaque métrique : comment la mesures-tu ? Quelle valeur cible vises-tu ?

### Exercice 2 — Construire un benchmark qualitatif (20 min)

Conçois un protocole d'évaluation humaine pour ta feature IA :
1. Échantillon de 20-50 outputs réels (ou simulés)
2. Grille d'évaluation (3-5 critères sur une échelle 1-3)
3. Process : qui évalue ? Comment ?
4. Seuil de qualité acceptable

### Mini-projet

**"Dashboard de métriques IA"**

Construis un tableau de bord (Notion, spreadsheet, ou wireframe) pour suivre une feature IA sur 90 jours. Il doit inclure :
1. Les métriques par niveau (technique, qualité, valeur, business)
2. Les seuils d'alerte pour chaque métrique
3. Un signal hebdomadaire (est-ce que ça va dans le bon sens ?)
4. Une section "verbatims" pour capturer les signaux qualitatifs

*Livrable : un tableau de bord documenté et justifié.*

## 8. Contraintes pédagogiques

Insister sur la complémentarité des métriques techniques et produit. Un PM ou designer qui ne voit que les métriques techniques est aveugle à la valeur réelle ; un data analyst qui ne voit que les métriques business manque les signaux de dégradation du modèle.
