# M2.7 — De la feature IA au produit IA-native

> Il y a une différence fondamentale entre un produit qui a ajouté une feature IA et un produit conçu dès le départ autour de l'IA. Ce module explore cette distinction et donne un cadre pour évaluer et transformer un produit existant.

---

## 1. Théorie

### Définition

Un **produit IA-native** est un produit dont l'architecture, l'UX, et la proposition de valeur centrale sont construites autour des capacités IA — pas une couche IA ajoutée sur un produit existant. Dans un produit IA-native, retirer l'IA ne laisse pas un produit dégradé, il laisse un produit fondamentalement non fonctionnel.

### Modèles mentaux

**Modèle 1 : Le spectre d'intégration IA**
```
Feature IA isolée          IA transversale          Produit IA-native
       │                         │                         │
"Bouton Résumer"          "IA dans 5 features"      "L'IA est le produit"
       │                         │                         │
Figma + Figma AI          Notion + Notion AI         Cursor, Perplexity
Retirer l'IA =            Retirer l'IA =             Retirer l'IA =
  produit complet           produit dégradé            produit vide
```

**Modèle 2 : Les 4 caractéristiques d'un produit IA-native**
1. **L'IA est dans le flux principal** : pas un bouton "Activer IA" dans un coin, mais dans chaque interaction centrale
2. **La valeur est impossible sans IA** : la proposition de valeur différenciante ne peut être délivrée sans l'IA
3. **L'interface est conçue pour l'incertitude** : l'UX anticipe et gère les cas d'erreur IA comme des états normaux
4. **Le produit apprend des utilisateurs** : les interactions alimentent une amélioration continue du modèle

### Schéma conceptuel

```
FRAMEWORK DE TRANSFORMATION : Feature IA → Produit IA-native

ÉTAPE 1 — Audit de l'intégration actuelle
  Questions : Où est l'IA dans le parcours utilisateur ?
  Quelle % de la valeur vient de l'IA ?
  Quels jobs-to-be-done l'IA adresse-t-elle ?

ÉTAPE 2 — Identification des opportunités de centralisation
  Questions : Quels jobs-to-be-done pourraient être PRINCIPALEMENT
  adressés par l'IA, pas juste assistés ?
  Qu'est-ce qui serait possible si l'IA était 10x meilleure ?

ÉTAPE 3 — Redesign du flux principal
  Action : Repenser le flux centré sur l'IA
  "L'utilisateur décrit son objectif → l'IA propose et exécute"
  vs "L'utilisateur clique partout → bouton IA en option"

ÉTAPE 4 — Nouvelles métriques IA-native
  Métriques classiques + métriques IA :
  Taux d'activation des features IA / Qualité des outputs /
  Rétention liée à l'IA / NPS avec vs sans IA

ÉTAPE 5 — Boucle d'apprentissage
  Concevoir comment le produit s'améliore avec l'usage
  Feedback → Données → Amélioration du modèle
```

### Analogie

"La différence entre une voiture avec GPS et une voiture autonome. Dans la première, le GPS est une feature additionnelle : retirer le GPS, tu as encore une voiture. Dans la seconde, retirer l'IA, tu n'as plus rien. Les produits IA-natifs sont conçus comme des voitures autonomes : l'IA n'est pas un équipement optionnel, elle est le moteur."

## 2. Compréhension

### 5 questions de vérification

1. Pourquoi une startup qui construit un produit IA-native from scratch a-t-elle un avantage sur un acteur établi qui "ajoute de l'IA" ?
2. Quels sont les risques de transformer trop rapidement un produit existant en produit IA-native ?
3. Comment décider quels jobs-to-be-done méritent d'être "nativifiés" avec l'IA ?
4. Qu'est-ce que le "cold start problem" dans un produit IA-native et comment le résoudre ?
5. Comment maintenir la confiance utilisateur dans un produit IA-native quand le modèle se dégrade ou fait une erreur visible ?

### Pièges courants

- **Transformer trop vite** : passer d'un produit stable à un produit IA-native nécessite une refonte profonde qui peut casser ce qui marchait bien. Transition progressive recommandée.
- **Oublier les utilisateurs non-IA** : certains segments ne veulent pas d'IA dans leur flux principal. Ne pas forcer la transition pour 100% des utilisateurs simultanément.
- **Dépendance à un seul fournisseur IA** : un produit IA-native est vulnérable si le modèle sous-jacent change ses conditions d'utilisation ou sa tarification.
- **Négliger la conformité et l'éthique** : plus l'IA est centrale, plus les questions de biais, de confidentialité, et de responsabilité sont critiques.

## 3. Application Produit / Design

### Cas d'usage concrets

**Cas 1 : Notion → Notion AI (transition progressive)**
Notion était un produit de prise de notes structuré. Notion AI a été ajouté comme feature transversale ("améliorer le texte", "résumer", "générer une page"). Ce n'est pas encore un produit IA-native — retirer Notion AI laisse Notion entier. Mais la direction est là : les nouvelles features s'appuient de plus en plus sur l'IA.

**Cas 2 : Cursor (IA-native from scratch)**
Cursor est un éditeur de code construit autour de l'IA : l'IA comprend la codebase entière, anticipe les prochaines actions, génère et révise du code en contexte. Retirer l'IA de Cursor laisse un éditeur de texte vide de sa valeur différenciante. C'est un produit IA-native.

**Cas 3 : Transformation d'un outil RH**
Un logiciel de recrutement classique (ATS) intègre progressivement l'IA : d'abord pour trier les CVs (feature IA), puis pour rédiger les offres (feature IA 2), puis pour analyser les entretiens (feature IA 3). À terme : un produit où le recruteur décrit le poste et les exigences, et l'IA prend en charge l'ensemble du pipeline de qualification.

### Traduction IA → stratégie produit

| Dimension | Produit avec feature IA | Produit IA-native |
|---|---|---|
| Proposition de valeur | "Notre produit + de l'IA" | "Seul possible grâce à l'IA" |
| Flow principal | Inchangé, IA en option | Redesigné autour de l'IA |
| Métriques | Adoption de la feature IA | Impossible sans IA |
| Compétition | Feature gap | Moat technologique |
| Risque modèle | Dégradation cosmétique | Impact central |

## 4. Pratique

### Exercice 1 — Audit d'un produit existant (30 min)

Prends un produit que tu connais bien. Évalue-le sur le spectre Feature IA / IA transversale / IA-native :
1. Où est l'IA dans le parcours utilisateur actuellement ?
2. Quelle % de la valeur vient de l'IA ?
3. Que resterait-il si l'IA était retirée demain ?
4. Quelle est la prochaine étape naturelle vers plus d'IA-native ?

### Exercice 2 — Imaginer la version IA-native (30 min)

Pour le même produit, imagine la version IA-native radicale :
- Quel job-to-be-done central serait entièrement adressé par l'IA ?
- Comment serait redessiné le flux principal ?
- Quelles features actuelles deviendraient redondantes ?
- Quels nouveaux risques apparaîtraient ?

### Mini-projet

**"Vision IA-native — Document de stratégie"**

Rédige un document de vision stratégique (2-3 pages) pour transformer un produit en direction IA-native :
1. Diagnostic actuel (audit du spectre d'intégration)
2. Vision IA-native à 18 mois
3. Roadmap de transition (phases progressives)
4. Métriques de succès IA-native
5. Risques et garde-fous
6. Impact sur l'équipe et les compétences requises

*Livrable : document de vision utilisable pour aligner leadership et équipe produit.*
