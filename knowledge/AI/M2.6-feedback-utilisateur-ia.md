# M2.6 — Concevoir un système de feedback utilisateur pour l'IA

> Un produit IA sans mécanisme de feedback est statique. Un produit avec un bon système de feedback s'améliore en continu. Ce module donne les outils pour concevoir des boucles de feedback qui améliorent réellement le modèle — pas juste qui collectent des données inutilisées.

---

## 1. Théorie

### Définition

Un système de feedback utilisateur pour l'IA est l'ensemble des mécanismes qui permettent de capturer les signaux d'évaluation des utilisateurs (explicites ou implicites), de les transformer en données d'entraînement, et de les utiliser pour améliorer les outputs du système. Le RLHF (Reinforcement Learning from Human Feedback) est la forme la plus sophistiquée de ce processus.

### Modèles mentaux

**Modèle 1 : Le spectre feedback explicite → implicite**
```
Explicite ←────────────────────────────→ Implicite
(demandé à l'utilisateur)           (observé dans le comportement)

Note 1-5    Pouce haut/bas    Correction    Taux d'acceptation    Taux d'abandon
───────────────────────────────────────────────────────────────────────────────
• Facile à    • Simple,        • Très        • Ne demande        • Révèle les
  interpréter   moins fiable     précis        rien à             cas extrêmes
• Faible        (gaming)       • Coûteux      l'utilisateur
  taux de      • Bon signal    • Rare        • Biais de
  réponse       d'insatisfact.               sur-acceptation
```

**Modèle 2 : La boucle d'amélioration continue**
```
Outputs IA → Feedback utilisateur → Données labellisées
     ↑                                        ↓
Modèle amélioré ← Fine-tuning / RLHF ← Analyse + filtrage
```
Chaque iteration améliore la qualité — à condition que le feedback soit de qualité, représentatif, et non biaisé.

### Schéma conceptuel

```
ARCHITECTURE D'UN SYSTÈME DE FEEDBACK IA

COUCHE 1 — Collecte (dans le produit)
  ├── Feedback explicite
  │   ├── Pouce haut/bas sur chaque output
  │   ├── Sélection parmi plusieurs variantes (meilleur que A/B)
  │   └── Correction directe (edit-in-place → log des modifications)
  │
  └── Feedback implicite
      ├── Taux d'acceptation sans modification
      ├── Taux de régénération
      ├── Taux d'abandon en cours de génération
      └── Usage en aval (l'output a-t-il été publié / envoyé / utilisé ?)

COUCHE 2 — Traitement (infrastructure)
  ├── Normalisation et déduplication
  ├── Filtrage du bruit (gaming, malveillance)
  ├── Stratification (s'assurer que tous les cas sont représentés)
  └── Annotation humaine des cas ambigus

COUCHE 3 — Amélioration (ML / Prompt engineering)
  ├── Mise à jour des prompts système (rapide, sans réentraînement)
  ├── Fine-tuning sur les exemples positifs (plus lourd)
  └── RLHF : modèle de récompense entraîné sur les préférences
```

### Analogie

"Un système de feedback IA, c'est comme le système de vote d'une plateforme collaborative : plus les utilisateurs votent pour les bonnes réponses et signalent les mauvaises, plus le contenu de qualité remonte. Sauf qu'ici, au lieu de trier du contenu, on entraîne un modèle à produire de meilleures réponses directement."

## 2. Compréhension

### 5 questions de vérification

1. Pourquoi le feedback explicite (pouce haut/bas) a-t-il un taux de participation faible et comment contourner ce problème ?
2. Comment le "gaming" du feedback (utilisateurs qui cliquent systématiquement sur "bon") biais-t-il le système ?
3. Qu'est-ce que le "preference learning" et comment diffère-t-il d'un système de notation classique ?
4. Comment s'assurer que le feedback collecté est représentatif de tous les utilisateurs, pas seulement des plus engagés ?
5. Dans quelle situation est-il préférable de NE PAS utiliser le feedback utilisateur pour améliorer le modèle ?

### Pièges courants

- **Feedback sans action** : collecter du feedback sans infrastructure pour l'utiliser génère de la frustration utilisateur et du coût sans bénéfice.
- **Sur-pondérer les utilisateurs très actifs** : les power users ont souvent des besoins atypiques — leur feedback peut dégrader l'expérience des utilisateurs majoritaires.
- **Ignorer le feedback négatif rare** : les cas d'erreurs graves (même rares) sont souvent les plus importants à corriger.
- **Feedback loop trop rapide** : mettre à jour le prompt ou le modèle trop fréquemment rend les comportements imprévisibles pour l'équipe.

## 3. Application Produit / Design

### Cas d'usage concrets

**Cas 1 : Chatbot de support — Boucle simple**
À la fin de chaque conversation, une question : "Cette réponse vous a-t-elle aidé ?" avec Oui/Non. Les "Non" déclenchent un tag dans Intercom + une notification à l'équipe. Chaque semaine, analyse des 20 pires conversations → ajustements du prompt système. Résultat sur 3 mois : taux de satisfaction +18%.

**Cas 2 : Assistant de rédaction — Feedback implicite riche**
Un outil de rédaction trace : % du texte IA conservé vs modifié, temps passé sur l'output, taux de régénération. Ces métriques, agrégées sur des milliers d'utilisateurs, révèlent les patterns d'échec (ex : les outputs sur le ton "formel" sont modifiés à 70% → le prompt "formel" est mal calibré).

**Cas 3 : Recommandation de contenu — Preference learning**
Au lieu de noter les recommandations 1-5, on montre à l'utilisateur 2 options et on lui demande laquelle il préfère. Ce signal de préférence comparative est plus fiable et utilisé directement pour entraîner un modèle de scoring.

### Traduction IA → produit

| Mécanisme de feedback | Signal collecté | Usage amélioration |
|---|---|---|
| Pouce haut/bas | Bon/mauvais output | Identifier les patterns d'échec |
| Edit-in-place | Diff avant/après | Données d'entraînement précises |
| Choix parmi variantes | Préférence comparative | Preference learning |
| Taux d'acceptation | Signal agrégé | Mesure de qualité globale |
| Régénération | Signal fort d'insatisfaction | Identifier les prompts défaillants |

## 4. Pratique

### Exercice 1 — Audit d'un système de feedback existant (25 min)

Analyse le système de feedback de ChatGPT, Claude, ou Perplexity :
- Quels mécanismes de feedback explicite sont présents ?
- Quels mécanismes de feedback implicite supposez-vous être trackés ?
- Que manque-t-il selon toi ?
- Comment ce feedback est-il probablement utilisé pour améliorer le modèle ?

### Exercice 2 — Concevoir les mécanismes de feedback (25 min)

Pour ta feature IA (issue de M2.3), conçois un système de feedback en 3 couches :
1. Feedback explicite : quels mécanismes ? À quelle fréquence les montrer pour ne pas fatiguer l'utilisateur ?
2. Feedback implicite : quels comportements tracer sans friction ?
3. Utilisation : comment l'équipe utilisera-t-elle ce feedback ? (ajustement prompt, fine-tuning, monitoring)

### Mini-projet

**"Roadmap de l'amélioration continue"**

Pour un produit IA existant ou imaginaire, conçois la roadmap de l'amélioration continue sur 6 mois :
- Mois 1-2 : mise en place de la collecte de feedback
- Mois 3-4 : premiers ajustements basés sur les données
- Mois 5-6 : évaluation de l'impact + décision sur le fine-tuning

*Livrable : document de roadmap avec jalons et indicateurs de succès.*
