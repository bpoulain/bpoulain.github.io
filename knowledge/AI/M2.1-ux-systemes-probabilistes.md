# M2.1 — Concevoir avec l'incertitude : l'UX des systèmes probabilistes

> Le plus grand défi du design de produits IA n'est pas technique — c'est de concevoir des interfaces qui fonctionnent bien quand le système peut toujours se tromper. Ce module pose les fondations du design pour les systèmes non-déterministes.

---

## 1. Théorie

### Définition

Un système probabiliste est un système dont les outputs ne sont pas garantis ni reproductibles à l'identique — contrairement à un bouton "Envoyer" qui envoie toujours un email. Un LLM peut produire une réponse différente au même prompt, et cette réponse peut être inexacte. Designer pour de tels systèmes requiert de repenser les conventions UX classiques basées sur la certitude.

### Modèles mentaux

**Modèle 1 : Designer pour un acteur compétent mais faillible**
Traite le système IA comme un collaborateur humain senior : compétent dans son domaine, mais qui peut se tromper, qui a des angles morts, et qui a besoin d'être supervisé sur les décisions critiques. Cette perspective évite deux erreurs : sur-confiance (déléguer sans vérification) et sous-utilisation (ne jamais faire confiance à l'IA).

**Modèle 2 : Le spectre confiance/contrôle**
Chaque feature IA se positionne sur un spectre entre deux extrêmes :
- **Tout déléguer** : l'IA décide et agit, l'humain voit le résultat (ex : filtres spam)
- **Tout contrôler** : l'IA suggère, l'humain valide chaque action (ex : suggestion d'email à envoyer)
La bonne position dépend du risque de l'erreur et de la fréquence d'utilisation.

### Schéma conceptuel

```
Risque d'une erreur IA
    ↑
    │  HAUTE  ─────────────────────────────────────────
    │          Validation humaine obligatoire avant action
    │          Afficher les sources et le niveau de confiance
    │          Permettre de modifier facilement
    │
    │  MOYENNE ─────────────────────────────────────────
    │          Suggestion visible avec option d'override
    │          Mécanisme de feedback (pouce haut/bas)
    │          Undo facile si erreur
    │
    │  FAIBLE  ─────────────────────────────────────────
    │          Autonomie possible (ex : tri, catégorisation)
    │          Notification de l'action réalisée
    └────────────────────────────────────────────────→
        Fréquence d'utilisation (rare → très fréquent)
```

### Analogie

"Concevoir l'UX d'un système IA, c'est comme designer un GPS : il peut se tromper, donc il te dit 'dans 300 mètres, tournez à droite' — et tu peux quand même décider d'aller tout droit si tu sais que la route est coupée. Il suggère, tu décides. Le bon design IA fonctionne de la même façon."

## 2. Compréhension

### 5 questions de vérification

1. Pourquoi les conventions UX classiques (ex : un bouton qui fait toujours la même chose) sont-elles inadaptées aux systèmes IA ?
2. Comment représenter visuellement l'incertitude d'un système IA sans anxiété ou méfiance excessive ?
3. Quelle est la différence entre un output IA "haute confiance" et "basse confiance" du point de vue UX ?
4. Dans quelle situation est-il acceptable de ne pas demander de confirmation à l'utilisateur avant une action IA ?
5. Comment concevoir un mécanisme de feedback utilisateur qui améliore le modèle sur le long terme ?

### Pièges courants

- **Masquer l'incertitude** : afficher un output IA avec la même certitude apparente qu'un résultat de recherche induit l'utilisateur en erreur.
- **Trop de friction** : demander une validation à chaque action IA détruit l'intérêt de l'automatisation.
- **"AI washing"** : présenter une feature comme "IA" sans que ça apporte de valeur différenciante génère de la méfiance quand elle échoue.
- **Négliger les états d'erreur** : les états "l'IA n'a pas pu répondre" ou "l'IA est incertaine" doivent être conçus autant que le happy path.

### Limites du concept

Ce module traite de l'UX des systèmes probabilistes en général. Il ne couvre pas les patterns d'interface spécifiques (voir M2.5) ni les mécanismes de feedback avancés (voir M2.6).

## 3. Application Produit / Design

### Cas d'usage concrets

**Cas 1 : Auto-complétion dans un éditeur de texte**
GitHub Copilot suggère du code en gris clair — pas en noir comme le code validé. L'utilisateur appuie sur Tab pour accepter, ou continue à taper pour ignorer. Le design communique : "c'est une suggestion, pas une vérité". Friction quasi-nulle, contrôle total.

**Cas 2 : Catégorisation automatique de dépenses**
Une app bancaire catégorise automatiquement les dépenses. Quand la confiance est haute (>95%), elle affiche directement la catégorie. Quand elle est basse, elle propose une suggestion avec un bouton "Confirmer ou changer". L'utilisateur ne voit la friction que quand c'est nécessaire.

**Cas 3 : Résumé automatique de réunion**
Notion/Otter AI génère un résumé. Le design présente clairement : "Résumé généré par IA — peut contenir des erreurs". Le texte est éditable directement. Un bouton "Régénérer" permet un nouvel essai. La responsabilité de la validation reste à l'humain.

### Traduction IA → produit

| Convention classique | Convention IA-aware |
|---|---|
| Output binaire (succès/échec) | Output avec niveau de confiance |
| Action irréversible immédiate | Action avec confirmation ou undo |
| Interface identique à chaque usage | Interface adaptative selon la confiance |
| L'erreur est une exception | L'erreur est un état normal prévu |

### Exemple d'intégration produit réel

Gmail Smart Reply propose 3 suggestions de réponse courtes. Chaque suggestion est présentée comme une option, pas une vérité. L'utilisateur peut en choisir une, la modifier, ou l'ignorer. Ce pattern de "suggestion légère avec contrôle total" est l'un des plus réussis en UX IA.

## 4. Pratique

### Exercice 1 — Audit d'une interface IA existante (20 min)

Choisis une application qui utilise de l'IA (Gmail, Grammarly, Spotify, etc.). Audite son interface avec ces critères :
1. Est-ce que l'utilisateur sait qu'il interagit avec de l'IA ?
2. Comment l'incertitude est-elle (ou n'est-elle pas) représentée visuellement ?
3. Quel est le mécanisme de correction si l'IA se trompe ?
4. Y a-t-il trop peu ou trop de friction dans les interactions IA ?
5. Une amélioration concrète que tu proposerais

### Exercice 2 — Concevoir les états d'erreur (15 min)

Pour une feature IA de ton choix, design les 4 états suivants (texte ou wireframe rapide) :
- État nominal (la réponse est bonne et confiante)
- État incertain (la réponse est probable mais pas sûre)
- État d'erreur silencieux (l'IA s'est trompée sans que l'utilisateur s'en rende compte)
- État de refus (l'IA ne peut pas répondre à cette question)

### Mini-projet

**"Redesign IA-aware d'une feature existante"**

Prends une feature non-IA dans un produit que tu connais (un moteur de recherche, un formulaire, une liste) et reimagine-la avec une couche IA. Documente :
1. La feature originale (avant)
2. La feature augmentée IA (après)
3. Les 3 décisions UX clés pour gérer l'incertitude
4. Les états d'erreur prévus
5. Le mécanisme de feedback utilisateur

*Livrable : un avant/après annoté (Figma, Miro, ou description textuelle structurée).*

## 8. Contraintes pédagogiques

Ce module présente des **principes**, pas des règles absolues. Chaque contexte produit requiert d'adapter le niveau de friction et de transparence. La règle d'or : le coût de l'erreur IA doit déterminer le niveau de supervision humaine.
