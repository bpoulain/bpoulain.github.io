# M3.2 — Automatisation vs augmentation : choisir le bon paradigme

> Automatiser ou augmenter ? Ce choix détermine l'architecture du système, le niveau de risque, et l'adoption par les équipes. Se tromper de paradigme est l'une des erreurs les plus coûteuses dans un projet IA.

---

## 1. Théorie

### Définition

**L'automatisation** retire l'humain de la boucle : l'IA effectue la tâche entière de façon autonome. **L'augmentation** garde l'humain dans la boucle : l'IA assiste, suggère, et accélère — mais l'humain décide et valide. Ces deux paradigmes ne sont pas en opposition, ils répondent à des besoins différents selon le contexte.

### Modèles mentaux

**Modèle 1 : La matrice Risque × Fréquence**
```
Risque d'erreur
    ↑
    │  ÉLEVÉ  ─────────────────────────────────────
    │          Augmentation obligatoire
    │          (humain valide avant toute action)
    │          Ex : diagnostic médical, décision juridique
    │
    │  MOYEN  ─────────────────────────────────────
    │          Augmentation avec option d'override
    │          (IA suggère, humain peut corriger)
    │          Ex : tri emails, catégorisation tickets
    │
    │  FAIBLE ─────────────────────────────────────
    │          Automatisation possible
    │          (IA agit, humain revoit en lot)
    │          Ex : filtres spam, formatage, traduction
    └────────────────────────────────────────────────→
        Fréquence (rare → très fréquente)
```

**Modèle 2 : Les 3 questions décisives**
1. **L'erreur est-elle réversible ?** Si une erreur est difficile à corriger (email envoyé, fichier supprimé, décision communiquée), l'humain doit être dans la boucle.
2. **La tâche implique-t-elle un jugement contextuel ?** Si oui, l'augmentation est préférable — l'IA fournit le contexte, l'humain le juge.
3. **Le volume justifie-t-il l'autonomie ?** Seulement si le volume est trop élevé pour une validation humaine systématique.

### Schéma conceptuel

```
Automatisation
┌────────────────────────────────────────────────────────┐
│  Input → IA → Action                                   │
│                                                        │
│  + Très rapide, scalable                               │
│  + Zéro effort humain                                  │
│  - Erreurs sans supervision                            │
│  - Adoption difficile si les erreurs sont visibles     │
│                                                        │
│  Exemples : filtre spam, resize automatique, tagging   │
└────────────────────────────────────────────────────────┘

Augmentation
┌────────────────────────────────────────────────────────┐
│  Input → IA → Suggestion → Humain valide → Action      │
│                                                        │
│  + Contrôle humain sur les décisions critiques         │
│  + Meilleure adoption (la personne reste "auteure")    │
│  + Les erreurs sont interceptées avant impact          │
│  - Plus lent que l'automatisation pure                 │
│  - Requiert une interface de validation bien conçue    │
│                                                        │
│  Exemples : suggestion d'email, auto-complétion code,  │
│             résumé à valider, tickets pré-remplis      │
└────────────────────────────────────────────────────────┘
```

### Analogie

"L'automatisation, c'est un Roomba : il nettoie sans que tu le surveilles, parfois il coince sous le canapé, mais ça reste acceptable. L'augmentation, c'est un GPS : il te dit quoi faire, mais c'est toi qui tiens le volant. Tu ne delègues pas la conduite — tu bénéficies d'une intelligence supplémentaire."

## 2. Compréhension

### 5 questions de vérification

1. Pourquoi l'automatisation complète n'est-elle pas toujours l'objectif à atteindre ?
2. Comment le coût d'une erreur influence-t-il le choix entre automatisation et augmentation ?
3. Qu'est-ce que le "human-in-the-loop" et dans quels cas est-il indispensable ?
4. Comment peut-on faire évoluer un système d'augmentation vers plus d'automatisation dans le temps ?
5. Pourquoi l'adoption par les équipes est-elle souvent meilleure avec l'augmentation qu'avec l'automatisation ?

### Pièges courants

- **Automatiser par défaut** : l'automatisation semble toujours plus "impressionnante", mais dans beaucoup de cas, l'augmentation crée plus de valeur et moins de risques.
- **Ignorer l'adoption** : un système d'automatisation rejeté par les équipes (parce qu'il fait trop d'erreurs ou remplace des tâches valorisées) est un échec même s'il est techniquement parfait.
- **"Full auto ou rien"** : les deux paradigmes peuvent coexister dans le même produit. Certaines sous-tâches s'automatisent, d'autres s'augmentent.
- **Négliger la supervision** : même un système automatisé nécessite une supervision régulière pour détecter les dérives de qualité.

### Limites du concept

La distinction automatisation/augmentation est un cadre de décision, pas une taxonomie rigide. En pratique, la plupart des systèmes IA combinent les deux. Ce module donne les critères de décision, pas une règle absolue.

## 3. Application Produit / Design

### Cas d'usage concrets

**Cas 1 : Triage des demandes support — Automatisation partielle**
Un outil support reçoit 500 tickets/jour. Les tickets "mot de passe oublié" (pattern très clair) sont automatiquement triés et une réponse-type est envoyée. Les tickets "je veux résilier" sont augmentés : l'IA rédige une réponse mais un humain valide avant envoi. Deux paradigmes coexistent selon le risque.

**Cas 2 : Génération de copy marketing — Augmentation**
Un outil de CRM permet aux commerciaux de générer des emails personnalisés en 1 clic. L'IA rédige, le commercial modifie et envoie. L'automatisation complète (IA envoie directement) a été rejetée par les équipes qui voulaient garder le contrôle sur leur relation client.

**Cas 3 : Monitoring de performance produit — Automatisation avec alertes**
Un dashboard analytique détecte automatiquement les anomalies (chute du taux de conversion, spike d'erreurs). Aucune action automatique — l'alerte est envoyée à l'équipe produit qui décide quoi faire. Automatisation de la détection, augmentation de la décision.

### Traduction IA → processus

| Question de décision | Automatisation | Augmentation |
|---|---|---|
| Réversibilité de l'erreur | Erreur réversible facilement | Erreur difficile à corriger |
| Volume | Très élevé (impossible à valider) | Gérable avec validation |
| Jugement contextuel requis | Non (pattern clair) | Oui (nuance nécessaire) |
| Acceptabilité de l'erreur | Faible impact | Impact visible ou fort |
| Adoption par l'équipe | Tâche peu valorisée | Tâche d'expertise |

### Exemple d'intégration produit réel

Grammarly est un cas exemplaire d'augmentation : il souligne les erreurs, propose des corrections, et suggère des améliorations de style — mais l'écrivain décide d'accepter ou non chaque suggestion. Si Grammarly corrigeait automatiquement sans montrer les changements, l'adoption serait beaucoup plus difficile.

## 4. Pratique

### Exercice 1 — Classifier 10 cas (20 min)

Pour chacune des tâches suivantes, indique : automatisation complète, automatisation partielle, augmentation, ou pas d'IA :

1. Corriger l'orthographe d'un email avant envoi
2. Envoyer un email de relance à un client inactif
3. Catégoriser des transactions bancaires
4. Décider d'accorder un crédit à un client
5. Résumer automatiquement une réunion et l'envoyer à tous les participants
6. Prioriser le backlog produit
7. Tagger des photos de produit sur un e-commerce
8. Rédiger un rapport médical à partir de notes
9. Router un ticket support vers le bon département
10. Générer des variantes A/B d'un titre de page

*Critère de réussite : chaque réponse est justifiée par un critère précis.*

### Exercice 2 — Analyser un cas réel (15 min)

Prends un processus que ton équipe vient de démarrer ou envisage d'automatiser. Applique les 3 questions décisives du modèle mental. Conclusion : automatisation, augmentation, ou hybride ? Argumente.

### Mini-projet

**"Matrice de décision pour mon équipe"**

Construis une matrice de décision réutilisable pour ton équipe. Elle doit permettre à n'importe quel membre de l'équipe de décider si un nouveau cas d'usage doit être automatisé ou augmenté. Format : tableau de 5 questions avec des réponses binaires (Oui/Non) qui mènent à une recommandation.

*Livrable : une matrice d'1 page, testée sur 3 cas concrets de ton équipe.*
