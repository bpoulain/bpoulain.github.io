# M1.3 — Le prompt comme interface : du texte à l'action

> Le prompt est la surface de contrôle d'un LLM. Comprendre comment le structurer te permet d'en extraire beaucoup plus, d'éviter les comportements erratiques, et de concevoir des features produit qui guident les utilisateurs vers des interactions efficaces.

---

## 1. Théorie

### Définition

Un prompt est l'ensemble du texte transmis à un LLM pour obtenir une réponse. C'est à la fois une interface utilisateur (ce que l'humain écrit) et une instruction machine (ce que le modèle interprète). Contrairement à une commande informatique classique, un prompt n'a pas de syntaxe stricte — mais il a des patterns qui fonctionnent et des anti-patterns qui échouent.

### Modèles mentaux

**Modèle 1 : Le prompt comme brief créatif**
Le modèle fonctionne comme un collaborateur qui a besoin d'un contexte clair pour produire un travail pertinent. Un brief vague produit un résultat vague. Un brief structuré (rôle + contexte + tâche + contraintes + format attendu) produit un résultat précis. *Conséquence design : traiter la conception de prompts comme la conception d'un brief, pas comme une commande informatique.*

**Modèle 2 : Le prompt comme espace de contraintes**
Chaque information ajoutée dans un prompt réduit l'espace des réponses possibles du modèle. Plus tu fournis de contexte pertinent, plus la réponse converge vers ce que tu veux. Trop peu de contexte → réponse générique. Trop de contexte non pertinent → le modèle peut se perdre.

### Schéma conceptuel

Structure d'un prompt efficace :

```
[RÔLE]
"Tu es un UX writer expert en applications B2B SaaS..."

[CONTEXTE]
"Nous lançons une feature de collaboration en temps réel pour des équipes de 5 à 50 personnes..."

[TÂCHE]
"Rédige 5 variantes de message d'onboarding pour le premier lancement de cette feature..."

[CONTRAINTES]
"Ton : professionnel mais chaleureux. Longueur : max 2 phrases par variante. Évite le jargon technique."

[FORMAT ATTENDU]
"Retourne un tableau avec : Variante | Ton perçu | Cas d'usage recommandé"
```

### Analogie

"Prompt = brief à un freelance qui ne te connaît pas. Plus tu lui donnes de contexte sur toi, sur le projet, sur ce que tu veux ET ce que tu ne veux pas, plus son livrable sera exploitable du premier coup."

## 2. Compréhension

### 5 questions de vérification

1. Quelle est la différence entre un prompt système (system prompt) et un prompt utilisateur ?
2. Pourquoi "fais-moi un résumé" donne souvent un mauvais résumé, et comment améliorer ce prompt ?
3. Qu'est-ce que le "few-shot prompting" et dans quel cas l'utiliser ?
4. Pourquoi donner un rôle au modèle ("tu es un expert en...") améliore souvent la qualité des réponses ?
5. Comment tester si un prompt est robuste ou fragile ?

### Pièges courants

- **Prompt trop court** : "résume cet article" sans préciser la longueur, le public cible, ou le format → résultat générique.
- **Sur-précision inutile** : ajouter des contraintes contradictoires ou non pertinentes dégrade la qualité.
- **Confondre le prompt et l'UX** : dans un produit, les utilisateurs ne voient pas le prompt système — il doit être conçu par le product team, pas par l'utilisateur.
- **Négliger la robustesse** : un prompt qui fonctionne une fois n'est pas un prompt de production. Il faut le tester avec des variations d'input.

### Limites du concept

Le prompting a des limites inhérentes : il ne peut pas compenser un modèle fondamentalement inadapté à la tâche, ne peut pas injecter de connaissances en temps réel (voir RAG, M1.4), et ne résout pas le problème des hallucinations (il peut les réduire, pas les éliminer).

## 3. Application Produit / Design

### Cas d'usage concrets

**Cas 1 : Génération de contenu marketing personnalisé**
Un outil de CRM utilise un prompt système qui inclut le profil du client (secteur, taille, historique) et génère des emails de relance personnalisés. Le commercial voit uniquement le résultat et peut le modifier avant envoi. Le prompt est invisible mais central.

**Cas 2 : Extraction structurée depuis des notes**
Un outil de prise de notes comme Notion AI reçoit des notes réunion en vrac et les reformate en : décisions prises / actions à faire / questions ouvertes. Le prompt système définit ce schéma de sortie — l'utilisateur n'a qu'à valider.

**Cas 3 : Chatbot de documentation produit**
Le prompt système dit : "Tu es l'assistant de [Produit]. Réponds uniquement à partir de la documentation fournie dans le contexte. Si tu ne sais pas, dis-le clairement. Ne mens jamais sur les fonctionnalités." — Ce prompt réduit drastiquement les hallucinations.

### Traduction IA → produit

| Niveau de prompting | Ce que l'utilisateur voit | Ce que le product team conçoit |
|---|---|---|
| Zéro-shot | Interface texte libre | Prompt système caché |
| Few-shot | Exemples dans l'UI | Exemples dans le prompt système |
| Chain-of-thought | "Pense étape par étape" | Instruction dans le prompt |
| Prompt template | Formulaire structuré | Template avec variables |

### Exemple d'intégration produit réel

Notion AI présente à l'utilisateur un menu simple ("Améliorer le texte", "Raccourcir", "Traduire"). Derrière chaque bouton se cache un prompt système précis que l'équipe Notion a conçu et testé. L'utilisateur n'écrit jamais de prompt — il choisit une intention, et le prompt est exécuté en arrière-plan.

## 4. Pratique

### Exercice 1 — Itération de prompt (20 min)

Prends cette tâche : *"Génère 3 titres pour un article de blog sur les erreurs à éviter quand on intègre de l'IA dans un produit."*

**Round 1** : Envoie ce prompt tel quel. Note le résultat.

**Round 2** : Ajoute un rôle + un contexte (ex : audience, ton de la marque). Note l'amélioration.

**Round 3** : Ajoute des contraintes de format (longueur, style, mots à éviter). Compare.

*Critère de réussite : tu peux identifier précisément quelle modification a le plus amélioré le résultat.*

### Exercice 2 — Bibliothèque de prompts personnelle (15 min)

Crée un tableau avec 5 prompts réutilisables pour tes cas d'usage courants (rédaction, synthèse, analyse, brainstorming, structuration). Pour chaque prompt :
- Contexte d'usage
- Prompt complet
- Exemple d'output obtenu
- Variantes testées

### Mini-projet

**"Prompt système pour un produit imaginaire"**

Conçois le prompt système d'un assistant IA intégré à un produit que tu connais (ou inventes). Il doit :
1. Définir le rôle et le périmètre de l'assistant
2. Lister les comportements autorisés et interdits
3. Spécifier le format de réponse attendu
4. Gérer le cas "je ne sais pas"
5. Refléter la voix et le ton de la marque

*Livrable : le prompt système complet, annoté avec les choix de conception.*

### Pour aller plus loin (profil technique)

La technique "Chain-of-Thought" (Wei et al., 2022) montre que demander au modèle de "raisonner étape par étape" améliore significativement les performances sur des tâches complexes. Implémentation : ajouter `"Réfléchis étape par étape avant de répondre."` à la fin du prompt.

## 8. Contraintes pédagogiques

Ce module suit M1.2 obligatoirement. Sans comprendre que le LLM prédit des tokens, il est impossible de comprendre *pourquoi* un prompt bien structuré fonctionne mieux.

Distinction clé à maintenir :
- **Concept** : le prompt comme espace de contraintes
- **Implémentation** : system prompt vs user prompt, température, max tokens
- **Usage produit** : l'utilisateur final ne voit jamais le prompt système — c'est une responsabilité design
